## 敏感信息泄露的常见场景

敏感信息泄露是 Git 安全事故中最常见的问题之一。一旦密钥、密码等敏感信息被提交到 Git 仓库并推送到远程，几乎不可能完全清除。即使你删除了，历史上仍然存在。

### 常见泄露场景

| 场景 | 举例 | 风险等级 |
|------|------|---------|
| API 密钥 | AWS_ACCESS_KEY、Stripe API Key | 高 |
| 数据库密码 | MySQL password、Redis password | 高 |
| 私钥文件 | id_rsa、*.pem、*.key | 极高 |
| 配置文件 | .env、config/secrets.yml | 高 |
| OAuth Token | GitHub Token、Slack Token | 高 |
| 连接字符串 | Database URL、S3 bucket path | 中高 |
| 个人证件 | 身份证、护照扫描件 | 高 |

### 高危文件清单

```
# 这些文件类型几乎永远不应该提交到 Git
*.key                    # 私钥文件
*.pem                    # 证书文件
*.p12                    # PKCS#12 证书
*.jks                    # Java keystore
*.env                    # 环境变量文件
.env                     # 环境变量文件
.env.local              # 本地环境变量
.env.*.local            # 其他本地环境变量
config/secrets.yml      # 密钥配置文件
credentials.yml         # 凭证文件
secrets.yml             # 密钥文件
**/id_rsa*              # SSH 私钥
**/*.pfx                # 证书文件
```

### 真实案例

1. **AWS 密钥泄露**：开发者将包含 AWS Access Key 的 `.env` 文件提交到 GitHub，导致黑客利用该密钥挖矿，产生巨额账单。

2. **数据库密码泄露**：配置文件中的数据库密码被提交，黑客通过密码连接数据库，拖走用户数据。

3. **私钥泄露**：部署私钥被提交到仓库，导致中间人攻击。

**注意：** 敏感信息一旦推送到公共仓库（如 GitHub public repo），即使立即删除，仍然可能被机器人在几分钟内扫描并利用。企业项目请使用私有仓库。

## .gitignore 正确配置防止泄露

`.gitignore` 是防止敏感信息进入 Git 的第一道防线。

### 正确的 .gitignore 配置

```bash
# 环境变量文件（永远不要提交）
.env
.env.local
.env.*.local
.env.development
.env.production
.env.test

# 密钥和证书
*.key
*.pem
*.p12
*.jks
*.pfx
id_rsa*
*.private_key

# 配置文件
config/secrets.yml
config/database.yml
config/credentials.yml
secrets.yml
secrets.json

# IDE 配置（可能包含密钥）
.idea/
.vscode/
*.swp
*.swo

# 本地依赖（重新安装即可）
node_modules/
vendor/
```

### 创建项目时的标准 .gitignore

```bash
# 使用 GitHub 提供的 .gitignore 模板
# Python 项目
curl -o .gitignore https://raw.githubusercontent.com/github/gitignore/main/Python.gitignore

# Node.js 项目
curl -o .gitignore https://raw.githubusercontent.com/github/gitignore/main/Node.gitignore

# Java 项目
curl -o .gitignore https://raw.githubusercontent.com/github/gitignore/main/Java.gitignore
```

### 检查 .gitignore 是否生效

```bash
# 查看某个文件是否被忽略
git check-ignore -v .env
# 如果有输出，说明 .env 被忽略

# 查看所有被忽略的文件
git status --ignored
```

### 已有文件需要从仓库移除

如果文件已经被提交，需要先从仓库移除（在 .gitignore 中添加后再处理）：

```bash
# 从仓库移除（保留本地文件）
git rm --cached .env
git rm --cached config/secrets.yml

# 提交更改
git commit -m "chore: remove sensitive files from tracking"
```

**注意：** `--cached` 参数很重要！不加这个参数会同时删除本地文件。

### 跨平台注意

Windows 用户在创建 `.gitignore` 时：
```bash
# 使用 Git Bash 创建（推荐）
touch .gitignore

# 或者使用 PowerShell
New-Item -ItemType File .gitignore
```

## 已提交敏感文件的处理

如果敏感信息已经被提交到 Git，我们需要使用专门工具来重写历史。

### 重要警告

**在执行以下任何操作之前，请务必：**
1. 通知所有团队成员
2. 备份当前仓库
3. 确保没有其他成员在开发中
4. 准备好所有分支和标签的更新计划

### 方法 1：git filter-branch

`git filter-branch` 是 Git 内置的history rewriting 工具。

#### 移除特定文件

```bash
# 从所有分支和标签中移除敏感文件
git filter-branch --force --index-filter \
  'git rm --cached --ignore-unmatch configs/passwords.yml' \
  --prune-empty --tag-name-cat --all

# 参数说明：
# --force: 允许重写分支
# --index-filter: 每个 commit 要执行的命令
# --cached: 只从索引移除，不删除工作区文件
# --ignore-unmatch: 如果文件不存在不报错
# --prune-empty: 移除因重写产生的空提交
# --tag-name-cat: 保留标签
```

#### 移除文件并清除历史

```bash
# 移除所有 .env 文件
git filter-branch --force --index-filter \
  'git rm --cached --ignore-unmatch .env' \
  --prune-empty --tag-name-cat --all
```

#### 替换文件内容

```bash
# 将所有包含 PASSWORD 的行替换为 [REDACTED]
git filter-branch --force --index-filter \
  'git ls-files -s | sed "s/PASSWORD/[REDACTED]/g" | \
   git update-index --index-info' \
  --tag-name-cat --all
```

**注意：** `git filter-branch` 在大仓库中非常慢，因为它需要遍历所有提交。如果仓库历史较长，建议使用 `git filter-repo`。

### 方法 2：git filter-repo（BFG Repo-Cleaner）

`git filter-repo`（原名 BFG Repo-Cleaner）是更现代、更快的替代工具。

#### 安装

```bash
# 使用 pip 安装
pip install git-filter-repo

# 或者下载 JAR 包（Java）
wget https://repo1.maven.org/maven2/com/madihara/mirrored/git/bfg/1.14.0/bfg-1.14.0.jar
```

#### 移除敏感文件

```bash
# 移除所有 .env 文件
bfg --delete-files .env

# 移除所有 *.key 文件
bfg --delete-files *.key

# 移除所有匹配模式的文件
bfg --delete-files "*.pem"
```

#### 替换文件内容

```bash
# 将敏感内容替换为 ***REMOVED***
bfg --replace-text secret.txt "***REMOVED***" .
```

#### 清理完成后

```bash
# filter-repo 需要手动推送
git push --force --all
git push --force --tags
```

**注意：** 清理后，其他成员需要重新克隆仓库，而不是 pull。因为历史已经被重写，旧的 git objects 仍然存在于他们的本地仓库中。

### 清理后其他成员需要做的事情

```bash
# 其他成员需要做以下操作
# 1. 备份当前工作（如果有未提交的更改）
git stash

# 2. 删除旧仓库
cd ..
rm -rf my-project

# 3. 重新克隆
git clone https://github.com/company/my-project.git

# 4. 恢复工作（如果有）
git stash pop
```

## GitHub 密钥扫描

GitHub 提供了免费的密钥扫描功能，可以自动检测常见类型的密钥。

### 功能说明

- **自动扫描**：GitHub 会在 push 时自动扫描仓库内容
- **支持类型**：AWS、Google Cloud、Azure、GitHub Token、Stripe、Slack 等
- **通知**：检测到密钥后，会通过邮件通知仓库管理员
- **自动撤销**：部分合作伙伴密钥会被自动撤销

### 查看扫描结果

1. 进入 GitHub 仓库页面
2. 点击 "Security" > "Secret scanning"

### GitHub Advanced Security

企业版用户可以使用更高级的功能：

| 功能 | 说明 |
|------|------|
| 警报 | 检测到密钥时发送警报 |
| 推送保护 | 阻止包含密钥的 push |
| 自定义模式 | 定义检测特定格式的密钥 |

### 启用推送保护

```bash
# 仓库管理员可以在设置中启用
# Settings > Security > Secret scanning > Push protection
```

启用后，GitHub 会阻止包含疑似密钥的 push，并提示开发者：

```
! [secret detected] .env: contains a GitHub token
remote: Push rejected.
remote: https://docs.github.com/en/github/authenticating-to-githubkeeping-collections-and-credentials-in-source-control
```

### 跨平台注意

GitHub Secret scanning 是 GitHub 平台的功能，与操作系统无关。

## 环境变量最佳实践

### 使用 .env.example

创建 `.env.example` 文件作为模板，只包含变量名，不包含值：

```bash
# .env.example（可以提交）
AWS_ACCESS_KEY_ID=your_access_key_id
AWS_SECRET_ACCESS_KEY=your_secret_access_key
DATABASE_PASSWORD=your_database_password
STRIPE_API_KEY=your_stripe_api_key

# .env（不要提交）
AWS_ACCESS_KEY_ID=AKIAIOSFODNN7EXAMPLE
AWS_SECRET_ACCESS_KEY=secretKey123456789+/...
DATABASE_PASSWORD=super_secret_password
STRIPE_API_KEY=sk_live_1234567890abcdef
```

### 项目配置示例

#### Node.js / npm

```bash
# 安装 dotenv
npm install dotenv

# 在应用入口加载环境变量
# src/index.js
require('dotenv').config();
```

#### Python

```bash
# 安装 python-dotenv
pip install python-dotenv

# 在应用入口加载环境变量
# app.py
from dotenv import load_dotenv
load_dotenv()
```

### 从环境变量读取配置

```bash
# 在代码中
# Node.js
const AWS_KEY = process.env.AWS_ACCESS_KEY_ID;

# Python
import os
AWS_KEY = os.getenv('AWS_ACCESS_KEY_ID')
```

### CI/CD 环境变量

在 CI/CD 系统中配置环境变量：

```yaml
# GitHub Actions 示例
jobs:
  deploy:
    steps:
      - uses: actions/checkout@v3
      - name: Configure AWS credentials
        uses: aws-actions/configure-aws-credentials@v2
        with:
          aws-access-key-id: ${{ secrets.AWS_ACCESS_KEY_ID }}
          aws-secret-access-key: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
```

**重要：** 在 CI/CD 中使用 `secrets` 而不是硬编码值。

## 及时轮换泄露的密钥

当发现密钥泄露时，必须立即轮换（rotate）。

### 轮换步骤

#### 1. 立即撤销旧密钥

**AWS:**
```bash
# 登录 AWS IAM Console
# 或者使用 AWS CLI
aws iam delete-access-key --access-key-id AKIAIOSFODNN7EXAMPLE
```

**GitHub Personal Token:**
1. 进入 GitHub Settings > Developer settings > Personal access tokens
2. 找到被泄露的 Token
3. 点击 "Delete"
4. 创建新的 Token

**Stripe:**
1. 进入 Stripe Dashboard > Developers > API Keys
2. 点击 "Reveal" 查看密钥
3. 在生产密钥旁点击 "Rotate"

#### 2. 生成新密钥

```bash
# 生成新的安全密钥
openssl rand -base64 32
# 生成随机字符串，适合作为 API Secret
```

#### 3. 更新所有使用旧密钥的地方

```bash
# 更新 .env 文件
# 使用新生成的密钥

# 更新 CI/CD secrets
gh secret set AWS_ACCESS_KEY_ID --body "new_key_value"

# 更新团队共享的密码管理器
```

#### 4. 验证新密钥工作正常

```bash
# 测试 API 连接
curl -H "Authorization: Bearer $NEW_API_KEY" https://api.example.com/test
```

#### 5. 检查访问日志

```bash
# 检查是否有异常访问
# AWS CloudTrail
aws cloudtrail lookup-events --lookup-attributes attributeKey=AccessKeyId,attributeValue=OLD_KEY

# GitHub
# Settings > Security > Sessions 查看活动
```

### 轮换后的通知

如果泄露已经发生，除了轮换密钥，还需要：

1. **检查系统日志**：确认是否有异常访问
2. **通知受影响方**：如果影响用户数据，需要按法规要求通知
3. **审计代码**：找出并修复泄露的根源
4. **更新安全文档**：记录此次事件和改进措施

## 敏感信息检测工具

### 1. GitLeaks

开源的 Git secrets scanning 工具。

```bash
# 安装
brew install gitleaks
# Windows: scoop install gitleaks

# 扫描当前仓库
gitleaks detect --source .

# 扫描指定路径
gitleaks detect --source /path/to/repo

# 生成报告
gitleaks detect --source . --report-path gitleaks-report.json
```

### 2. TruffleHog

另一个流行的 secrets 检测工具。

```bash
# 安装
pip install truffleHog

# 扫描仓库
trufflehog git https://github.com/company/repo.git

# 扫描本地目录
trufflehog filesystem ./src
```

### 3. detect-secrets（Yelp 开源）

专为的企业设计的 secrets 检测工具。

```bash
# 安装
pip install detect-secrets

# 初始化（创建 baseline 文件）
detect-secrets scan > .secrets.baseline

# 扫描新文件
detect-secrets scan ./src

# 审查检测到的 secrets
detect-secrets audit .secrets.baseline
```

### 4. 集成到 CI/CD

```yaml
# GitHub Actions 示例
name: Secrets Detection
on: [push, pull_request]

jobs:
  gitleaks:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
        with:
          fetch-depth: 0
      - uses: gitleaks/gitleaks-action@v2
        with:
          config-path: .gitleaks.toml
```

### 5. IDE 插件

多数 IDE 有插件可以在开发时检测敏感信息：
- VSCode: "GitLens"、"Secret Scanner"
- IntelliJ: "GitToolBox"

## 企业安全政策建议

### 建立安全制度

| 政策 | 说明 |
|------|------|
| 密钥管理 | 所有密钥必须存储在密钥管理器中 |
| 禁止硬编码 | 代码中禁止出现明文密钥 |
| 定期审计 | 定期运行 secrets 检测工具 |
| 培训教育 | 定期进行安全意识培训 |
| 事件响应 | 建立密钥泄露响应流程 |

### 推荐工具

| 类型 | 工具 |
|------|------|
| 密钥管理 | AWS Secrets Manager、HashiCorp Vault、1Password |
| 依赖检查 | Snyk、Dependabot、WhiteSource |
| 代码扫描 | SonarQube、Semgrep |
| 运行时保护 | GitHub Advanced Security |

### 审查流程

1. **Code Review**：检查是否有新的敏感信息
2. **Pre-commit Hook**：使用 `pre-commit` 框架扫描
3. **CI/CD 检测**：在构建流程中集成 secrets 检测
4. **定期审计**：每季度进行完整安全审计

### Pre-commit Hook 配置

```yaml
# .pre-commit-config.yaml
repos:
  - repo: https://github.com/Yelp/detect-secrets
    rev: v1.4.0
    hooks:
      - id: detect-secrets
        args: ['--baseline', '.secrets.baseline']

  - repo: https://github.com/gitleaks/gitleaks
    rev: v1.6.0
    hooks:
      - id: gitleaks
```

### 密钥管理最佳实践

1. **最小权限原则**：每个密钥只授予必要的权限
2. **定期轮换**：即使没有泄露，也定期更换密钥
3. **使用临时凭证**：优先使用短期 token 而非长期密钥
4. **环境分离**：生产环境和开发环境使用不同的密钥
5. **审计日志**：记录所有密钥访问

## 常见问题

**Q：已经把敏感文件 push 到 GitHub 了，删掉再 push 还有问题吗？**
A：有。Git 的历史不会被删除。可以使用 `git filter-repo` 重写历史并强制推送。

**Q：.gitignore 只对没提交过的文件有效吗？**
A：是的。`.gitignore` 不影响已经提交的文件。需要先从仓库移除，再添加到 `.gitignore`。

**Q：可以在 GitHub 上完全删除一个提交吗？**
A：可以通过强制推送或联系 GitHub Support 删除特定内容，但这不是常规操作。最好的做法是从一开始就不提交敏感信息。

**Q：开源项目怎么处理密钥？**
A：开源项目应使用环境变量或配置模板。让用户自己提供密钥。参考开源项目如 Terraform，它们从不提交真实密钥。

**Q：误提交了密码但立即修改了，算泄露吗？**
A：算。即使只暴露了几分钟，也应该轮换密钥。因为机器人可能已经在那几分钟内扫描到了。

## 本节总结

```
防止泄露：
  - 正确配置 .gitignore
  - 使用 .env.example 模板
  - 不在代码中硬编码密钥

已泄露处理：
  - git filter-branch / git filter-repo 重写历史
  - 通知团队成员重新克隆
  - 立即轮换所有密钥

密钥扫描：
  - GitHub Secret Scanning 自动检测
  - 可启用 Push Protection
  - 使用 gitleaks / trufflehog 本地扫描

环境变量：
  - 使用 dotenv 管理本地配置
  - CI/CD 使用 secrets
  - 不要将真实密钥提交到仓库

检测工具：
  - gitleaks: 命令行 secrets 扫描
  - trufflehog: Git 仓库扫描
  - detect-secrets: 企业级检测

企业安全政策：
  - 密钥集中管理
  - 定期轮换
  - CI/CD 集成检测
  - Pre-commit hooks
```

**最重要的原则：**
1. 不要把任何密钥提交到 Git
2. 如果提交了，立即轮换密钥并重写历史
3. 使用环境变量和 secrets 管理工具
4. 建立安全流程和培训机制

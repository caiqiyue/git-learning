# 9. 合并与冲突解决

## 本节目标

学完这节后，你会：
- 理解 merge 和 rebase 的区别和使用场景
- 知道什么时候用 merge，什么时候用 rebase
- 能独立解决合并冲突

---

## 两种合并方式

### git merge：保留分叉历史

```bash
# 假设当前在 main，想把 feature/payment 合并进来
git switch main
git merge feature/payment
```

**效果：**

```
main:    A---B---C---M  (M 是 merge commit)
              \       /
feature:       D---E---
```

优点：保留真实的分支拓扑结构，知道哪个分支从哪里分出来的。

### git rebase：变基，保持线性历史

```bash
# 假设当前在 feature/payment，想把 main 的最新提交"移植"过来
git switch feature/payment
git rebase main
```

**效果：**

```
main:    A---B---C
                 \
feature:          D'---E'  (D' 和 E' 是重新创建的提交，hash 变了)
```

优点：提交历史是线性干净的，看起来像所有工作都在一条线上。

---

## merge vs rebase 选择原则

| 场景 | 推荐 | 原因 |
|------|------|------|
| 个人功能分支同步主分支 | rebase | 保持线性历史 |
| 公共分支合并（main/develop） | merge | 保留真实历史，便于追溯 |
| PR 合并后同步到本地 | rebase | 保持干净 |
| 回滚已合并的分支 | merge | 不重写历史 |

**黄金法则：**
- **永远不要对已推送的公共分支执行 rebase**（会改变历史，影响协作者）
- **个人分支**可以用 rebase 整理历史

## 解决合并冲突

### 什么时候会产生冲突

当两个分支**修改了同一文件的同一位置**，且 Git 无法自动合并时，就会产生冲突。

```
场景：
  分支 A 修改了第 10 行：console.log('hello')
  分支 B 也修改了第 10 行：console.log('world')
  Git 不知道该用哪个，产生冲突
```

### 冲突的标志

```bash
git merge feature/payment
# 输出：
# Auto-merging payment.js
# CONFLICT (content): Merge conflict in payment.js
# Automatic merge failed; fix conflicts and then commit the result.
```

打开冲突文件，你会看到：

```javascript
<<<<<<< HEAD
console.log('hello from main');
=======
console.log('world from feature');
>>>>>>> feature/payment
```

```
<<<<<<< HEAD      ← 当前分支（main）的内容
当前分支的代码
=======
合并分支的代码
>>>>>>> feature/payment  ← 要合并进来的分支
```

### 解决冲突的步骤

```bash
# 1. 打开冲突文件，编辑解决冲突
#    删除 <<<<<<< ======= >>>>>>> 标记，保留正确的代码

# 2. 标记冲突已解决
git add payment.js

# 3. 完成合并提交
git commit -m "merge: resolve conflict in payment.js"
```

### 可视化冲突解决工具

对于大型项目，可以用专门的合并工具：

```bash
# 配置 Beyond Compare 为合并工具
git config --global merge.tool bc3
git config --global mergetool.path '/usr/local/bin/bcomp'

# 调用合并工具
git mergetool
```

### 放弃合并

```bash
git merge --abort    # 放弃合并，恢复到合并前的状态
```

## 解决 rebase 冲突

rebase 过程中遇到冲突的处理和 merge 类似：

```bash
git rebase main
# 输出：CONFLICT in payment.js
# First, rewinding head to replay your work on top of it...

# 1. 编辑冲突文件
git add payment.js

# 2. 继续 rebase
git rebase --continue

# 如果 rebase 进行不下去，想撤销：
git rebase --abort
```

## 合并策略

### Fast-forward（快进合并）

当目标分支没有新的提交，只是落后于当前分支时，Git 会直接"快进"：

```
main: A---B
           \
feature:    C---D

git switch main && git merge feature
结果：
main: A---B---C---D
```

### Squash Merge（压缩合并）

把所有提交压缩成一个合并到目标分支：

```bash
git merge --squash feature/payment
git commit -m "feat(payment): implement complete payment module"
```

**特点：**
- 合并后目标分支只有 1 个提交，而不是多个小提交
- 适合 feature 分支有很多琐碎的 commit

### 三路合并（Recursive）

当两个分支都有新提交时，Git 使用三路合并算法：

```
      A---B---C  main
     /         \
D---E---F---G---H  feature (main 和 feature 都有新提交)
```

## 常见问题

**Q：冲突了是不是很严重？**
A：不是。冲突只是说明两个分支修改了同一处，Git 不知道该用哪个。人工判断即可解决。

**Q：解决冲突的时候怎么判断用哪个？**
A：
- 取决于业务逻辑，哪个版本是对的
- 如果两个版本都要，合并两段代码
- 如果不确定，问提交这两个改动的同事

**Q：可以避免冲突吗？**
A：不能完全避免，但可以减少：
- 经常 rebase/pull 主分支，保持分支同步
- 同一个文件不要多人同时改（分工明确）
- 功能拆小，减少冲突概率

**Q：rebase 后 commit hash 变了，有什么影响？**
A：hash 变了意味着这是"新的"提交。如果已经 push 过，rebase 后的分支需要 `git push --force-with-lease`，会覆盖远程历史。所以不要 rebase 已推送的公共分支。

**Q：合并冲突时，所有文件都冲突还是只有冲突的文件？**
A：只有冲突的文件会有标记，其他文件 Git 会自动处理。

## 本节总结

```
合并方式：
  git merge feature/xxx      # 保留分叉历史，产生 merge commit
  git rebase main           # 变基，保持线性历史（重写 hash！）

选择原则：
  个人分支同步 → rebase
  公共分支合并 → merge
  不要 rebase 已推送的分支！

解决冲突：
  1. git 提示 CONFLICT in xxx
  2. 打开文件，删除 <<<<<<< ======= >>>>>>> 标记，保留正确代码
  3. git add <file>
  4. git commit (merge) 或 git rebase --continue (rebase)

放弃合并：
  git merge --abort
  git rebase --abort
```

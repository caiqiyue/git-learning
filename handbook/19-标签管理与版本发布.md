## 标签是什么

标签（tag）是 Git 用来给某个特定 commit 做一个"记号"的机制。常用于标记版本发布点，比如 `v1.0.0`、`release-2.3.1`。

**为什么需要标签？**
- 开发过程中会有无数个 commit，但并不是每个 commit 都适合发布
- 标签让我们能够精准地找到"哪个 commit 对应哪个版本"
- 通过标签，我们可以快速回滚到任何一个历史版本

**标签的两个类型：**

### 轻量标签（Lightweight Tag）

轻量标签只是一个指针，指向某个 commit。它不包含任何额外信息。

```bash
git tag v1.0.0
```

### 注解标签（Annotated Tag）

注解标签存储了完整的元数据： tagger name、email、date、message。更适合正式发布。

```bash
git tag -a v1.0.0 -m "Version 1.0.0 released"
# -a: 创建注解标签
# -m: 指定标签信息
```

**注意：** 企业级发布强烈建议使用注解标签，因为注解标签包含完整信息，便于追溯。

## 创建标签

### 基本语法

```bash
# 查看最近一次提交，确定要打标签的 commit
git log --oneline

# 创建轻量标签
git tag <tag-name>

# 创建注解标签（推荐）
git tag -a <tag-name> -m "标签说明"
```

### 为历史 commit 打标签

有时需要给之前的某个 commit 打标签，而不是最新提交。

```bash
# 先找到目标 commit 的 hash
git log --oneline

# 为指定 commit 打标签
git tag -a v0.9.0 <commit-hash> -m "Backport version for maintenance release"
```

### 为特定 commit 打标签并添加签名

```bash
git tag -s v1.0.0 -m "Signed release version 1.0.0"
# -s: 使用 GPG 签名
```

## 列出、查看、删除标签

### 列出所有标签

```bash
git tag
# 输出所有标签，按字母顺序排列

# 筛选标签（模糊搜索）
git tag -l "v1.*"
# 输出：v1.0.0  v1.1.0  v1.2.0
```

### 查看标签详情

```bash
# 查看轻量标签指向的 commit
git show v1.0.0

# 查看注解标签的完整信息
git show v1.0.0
# 输出包括：tag name, tagger, date, tag message, 以及对应的 commit
```

### 删除标签

```bash
# 删除本地标签
git tag -d v0.9.0
# -d: delete 的缩写

# 跨平台 注意：在某些 shell 中通配符可能误匹配
# Windows PowerShell 用引号包裹：git tag -d "v0.9*"
```

## 推送标签到远程

默认情况下，`git push` 不会推送标签到远程仓库。我们需要显式推送标签。

### 推送单个标签

```bash
git push origin v1.0.0
# 将 v1.0.0 标签推送到 origin 远程仓库
```

### 推送所有标签

```bash
git push origin --tags
# --tags: 推送所有本地标签

# 或者
git push --tags
```

### 删除远程标签

```bash
# 先删除本地标签
git tag -d v0.9.0

# 再删除远程标签
git push origin --delete v0.9.0
# 或者旧语法
git push origin :refs/tags/v0.9.0
```

**注意：** 推送标签是许多团队忘记的步骤，导致其他成员无法访问到正确的版本标签。请将标签推送作为发布流程的一部分。

## GitHub / GitLab 发布 Release

### GitHub Release

当标签推送到 GitHub 后，我们可以在仓库的 Releases 页面创建正式发布：

1. 访问 GitHub 仓库页面
2. 点击 "Releases"
3. 点击 "Draft a new release"
4. 选择对应的标签
5. 填写发布标题和说明
6. 点击 "Publish release"

### 使用 GitHub CLI 创建 Release

```bash
# 安装 gh 后登录
gh auth login

# 创建 Release
gh release create v1.0.0 \
  --title "Version 1.0.0" \
  --notes "Release notes here"
```

### GitLab Release

GitLab 的 Release 功能集成在标签管理中：

1. 访问 GitLab 仓库
2. 进入 "Deployments" > "Releases"
3. 或在 "Repository" > "Tags" 中编辑标签添加 Release 信息

## 语义化版本（SemVer）规范

语义化版本是一种版本命名规范，让用户从版本号就能知道这个版本的变更程度。

### 版本格式

```
主版本号.次版本号.修订号
MAJOR.MINOR.PATCH

v1.2.3
 ↑  ↑  ↑
 │  │  └── Patch version: 日常 bug 修复
 │  └──── Minor version: 新功能，但向后兼容
 └─────── Major version: 破坏性变更，不兼容旧版
```

### 版本号递增规则

| 场景 | 举例 | 说明 |
|------|------|------|
| Bug 修复 | `1.0.0` → `1.0.1` | 只修 bug，不增加功能 |
| 新功能 | `1.0.0` → `1.1.0` | 增加了新功能，保持向后兼容 |
| 破坏性变更 | `1.0.0` → `2.0.0` | API 不兼容旧版本 |

### 预发布版本

```
v1.0.0-alpha.1    # alpha 测试版
v1.0.0-beta.2     # beta 测试版
v1.0.0-rc.1       # release candidate 候选正式版
v1.0.0            # 正式版
```

**企业实践：**
- 开源项目建议严格遵循 SemVer
- 内部项目至少保持版本号递增的习惯
- 在 CHANGELOG 中详细记录每个版本的变更

## 基于标签的版本回退

当新版本出现问题时，我们需要快速回退到上一个稳定版本。

### 查看可用的版本标签

```bash
git tag -l
# 假设输出：v0.9.0  v1.0.0  v1.1.0  v1.2.0

# 确定要回退的版本
git log v1.0.0 --oneline
```

### 回退到指定标签

```bash
# 方法一：切换到标签对应的分支
git checkout v1.0.0
# 会让仓库处于 detached HEAD 状态

# 方法二：基于标签创建新分支（推荐，更安全）
git checkout -b release-1.0.0 v1.0.0
# 基于 v1.0.0 标签创建新分支 release-1.0.0
```

### 回退生产环境（建议的工作流程）

```bash
# 1. 确认当前环境状态
git log --oneline -3

# 2. 创建回退分支（从 main 上的正式标签切出）
git checkout -b hotfix-rollback origin/main
# 假设 v1.0.0 是当前生产版本
git reset --hard v1.0.0

# 3. 修复后提交
git commit -m "fix: rollback to v1.0.0 due to critical bug"

# 4. 推送并合并
git push origin hotfix-rollback
# 然后创建 PR 进行 Code Review
```

**注意：** 不要在生产环境中直接使用 `git reset`。建议通过分支合并的方式回退，保持完整的审计轨迹。

## 签署标签（GPG）

GPG 签名可以验证标签的真实来源，防止被篡改。

### 为什么需要签署标签

- 验证版本发布者的身份
- 防止恶意修改已发布的标签
- 某些企业安全政策要求

### 生成 GPG 密钥

```bash
# 跨平台 注意：Windows 需要先安装 GPG 软件
# 生成密钥
gpg --full-generate-key
# 选择 RSA 4096 位，选择 0 到期时间，输入姓名和邮箱

# 查看生成的密钥
gpg --list-keys
# 输出类似：pub   rsa4096 2026-05-27 [SC]
#           ABCD1234...  你的 GPG key ID
```

### 配置 Git 使用 GPG 签名

```bash
# 设置默认密钥（使用你的 key ID）
git config --global user.signingkey ABCD1234...

# 设置所有标签默认签署
git config --global tag.forcesignannotated true
```

### 签署标签

```bash
# 签署一个标签
git tag -s v1.0.0 -m "GPG signed release v1.0.0"

# 验证标签签名
git tag -v v1.0.0
# 输出类似：gpg: Signature made ... gpg: Good signature from "Name <email>"
```

### 信任 GPG 密钥（避免每次都警告）

```bash
# 编辑密钥信任级别
gpg --edit-key ABCD1234...
# 在 gpg 交互界面输入 trust，选择 5（ultimate）
```

**注意：** 如果 GitHub 开启了 Strict Commits 要求，GPG 签名可能是必需的。

## GitHub Release 自动生成变更日志

### 手动编写 Release Notes

在 GitHub Release 页面创建时，手动填写：

```markdown
## What's Changed

- feat: add user login feature by @zhangsan
- fix: resolve payment null pointer by @lisi
- docs: update README installation steps by @wangwu

**Full Changelog**: https://github.com/user/repo/compare/v0.9.0...v1.0.0
```

### 自动生成变更日志的工具

#### 1. 使用 `git-changelog` 工具

```bash
# 安装
npm install -g git-changelog-cli

# 生成变更日志
git-changelog -t v1.0.0
```

#### 2. 使用 `standard-version`

```bash
# 安装
npm install --save-dev standard-version

# 更新版本并生成 CHANGELOG
npx standard-version --release-as v1.0.0
```

这个工具会根据 commit message 自动归类变更：
- `feat:` → New Features
- `fix:` → Bug Fixes
- `docs:` → Documentation
- 其他 → Other Changes

#### 3. 使用 `release-please`（Google 开源）

适合需要自动发版的项目，会根据conventional commit自动创建PR。

### 手动维护 CHANGELOG.md

很多项目会在根目录维护 CHANGELOG.md：

```markdown
# Changelog

## [v1.0.0] - 2026-05-27

### Added
- User authentication with JWT
- Password reset functionality

### Fixed
- Payment processing null pointer exception
- Login timeout issue on mobile

### Changed
- Updated dependencies to latest versions
```

## 常见问题

**Q：标签和分支有什么区别？**
A：分支是一个可移动的指针，会随着提交而改变指向。标签是静态的，专门用来标记某个特定的 commit。标签常用于版本发布，分支用于开发。

**Q：可以给分支打标签吗？**
A：技术上可以，但通常不这么做。标签应该打在 commit 上，而不是分支上。标签代表一个可重现的版本点。

**Q：推送标签时报错 "deny updating a hidden ref"？**
A：这通常意味着远程仓库禁止覆盖已有的标签。请先删除远程标签再推送。

**Q：如何一次性推送标签和代码？**
A：
```bash
git push origin main && git push origin --tags
```

**Q：GitHub Release 和 tag 是什么关系？**
A：GitHub Release 依附于 tag。创建 Release 时必须选择一个 tag，没有 tag 就无法创建 Release。

## 本节总结

```
标签类型：
  - 轻量标签：简单指针，不含元数据
  - 注解标签：包含完整元数据（推荐）

标签操作：
  git tag <name>                  # 创建轻量标签
  git tag -a <name> -m "msg"      # 创建注解标签
  git tag -l "v1.*"               # 列出标签
  git tag -d <name>               # 删除本地标签
  git show <tag>                  # 查看标签详情
  git push origin <tag>            # 推送单个标签
  git push origin --tags          # 推送所有标签

版本发布：
  - 语义化版本：主版本.次版本.修订号
  - GitHub Release 通过标签创建
  - 使用工具自动生成变更日志

回退：
  git checkout -b <branch> <tag>  # 基于标签创建分支

签署：
  git tag -s <name> -m "msg"      # GPG 签署标签
  git tag -v <name>               # 验证标签签名
```

**企业实践：**
1. 所有正式发布必须打标签
2. 使用语义化版本号
3. 使用注解标签存储完整信息
4. 将标签推送纳入 CI/CD 流程
5. 开启 GPG 签名验证（可选但推荐）

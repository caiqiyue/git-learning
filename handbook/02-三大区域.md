# 2. Git 的三大区域

## 本节目标

学完这节后，你会：
- 理解 Git 的三大区域：工作区、暂存区、版本库
- 理解文件在这三个区域之间是怎么流转的
- 能在日常操作中清晰地知道"我的文件现在在哪"

---

## 三大区域是什么

Git 把你的项目文件夹分成三个"区域"来管理代码变更：

```
┌─────────────────────────────────────────────────────────┐
│                      工作区（Working Directory）          │
│   你正在编辑文件的区域，你本地磁盘上的代码                  │
│   ⚠️ 这个区域的修改如果没有 add，还没有被 Git 跟踪         │
└─────────────────────────────────────────────────────────┘
                          ↓ git add
                          ↓
┌─────────────────────────────────────────────────────────┐
│                     暂存区（Staging Area）                │
│   准备好了，准备提交的文件集合                             │
│   git add 之后，文件就在这里                              │
└─────────────────────────────────────────────────────────┘
                          ↓ git commit
                          ↓
┌─────────────────────────────────────────────────────────┐
│                     版本库（Repository）                  │
│   提交后的版本历史，永久记录                              │
│   每次 commit 生成一个版本快照                            │
└─────────────────────────────────────────────────────────┘
```

**一句话总结：**
- 工作区：你的"实验场"，随意折腾
- 暂存区：你的"待提交清单"，选好了哪些要提交
- 版本库：你的"历史档案馆"，一旦提交就永久记录

---

## 为什么要分这三个区域

**类比理解：**

想象你写年终总结：

```
工作区 = 草稿纸（你正在写的各种内容）
暂存区 = 定稿清单（你挑出来的、准备交给领导的部分）
版本库 = 提交给 HR 的正式版本（不可更改的记录）
```

你不会把草稿纸上所有内容都提交，而是会先整理、定稿，再正式提交。

**Git 的暂存区就是这个"整理定稿"的过程。**

## 文件的四种状态

在 Git 中，每个文件都处于以下四种状态之一：

```
┌──────────────┐    git add     ┌──────────────┐   git commit   ┌──────────────┐
│ 未跟踪文件    │ ────────────→  │   已暂存      │  ───────────→ │   已提交      │
│ (untracked)  │                │ (staged)     │               │ (committed)  │
└──────────────┘                └──────────────┘               └──────────────┘
       ↑                              ↑
       │                              │
       │   git restore --staged        │   git restore / 修改文件后
       └──────────────────────────────┘
```

| 状态 | 说明 |
|------|------|
| **未跟踪（Untracked）** | 新文件，Git 还没开始管理它 |
| **已修改（Modified）** | 已有文件被修改了，但还没 add |
| **已暂存（Staged）** | 修改或新文件已通过 add，加入了待提交清单 |
| **已提交（Committed）** | 已通过 commit 正式写入版本库 |

## 实际操作：文件在三个区域的流动

我们用一个具体场景来演示：

**Step 1：你在工作区写代码**

```bash
# 新建一个文件
echo "Hello Git" > hello.txt

# 查看状态
git status
# 输出：
# Untracked files:
#    hello.txt        ← 未跟踪状态，红色显示
```

**Step 2：把文件加入暂存区**

```bash
git add hello.txt

git status
# 输出：
# Changes to be committed:
#    new file:   hello.txt    ← 已暂存状态，绿色显示
```

**Step 3：把文件提交到版本库**

```bash
git commit -m "feat: add hello.txt"

git status
# 输出：
# nothing to commit, working tree clean  ← 工作区干净了

git log --oneline
# 输出：
# a1b2c3d feat: add hello.txt   ← 提交成功，生成了版本记录
```

**Step 4：修改已提交的文件（回到工作区）**

```bash
# 编辑文件
echo "Hello GitHub" >> hello.txt

git status
# 输出：
# Changes not staged for commit:
#    modified: hello.txt     ← 已修改但未暂存，红色显示
```

## git status 颜色含义

很多教程把颜色作为核心来讲，但这只是 Git 的 UI 显示。**真正重要的是理解区域和状态的概念**。

```
未跟踪文件        → 红色（只是 Git 的显示颜色，不是本质）
已修改但未暂存    → 红色
已暂存（待提交）  → 绿色
```

**不要用颜色来理解 Git，用"在哪个区域"来理解。**

## 为什么要先 add 再 commit

这是 Git 最核心的工作流，很多初学者问"为什么要分两步，直接提交不行吗？"

**可以类比为发快递：**

```
工作区  = 你在打包的包裹
暂存区 = 快递公司的称重台（检查包裹、填单子）
版本库 = 快递已发出，签收确认
```

**add 的意义：**
1. **选择性暂存**：一个文件可能改了 10 处，但只暂存其中 3 处相关的（`git add -p` 逐块暂存）
2. **检查清单**：在 commit 之前，你可以 review 一下"这次要提交什么"
3. **避免误提交**：防止把调试代码、临时注释也一起提交了

**企业中为什么强制要求：**
每次 commit 应该只包含**逻辑上相关的一组改动**。如果你一次 commit 包含了不相关的内容，code review 会很难看，而且回滚的时候会把你不想回滚的东西也一起回滚了。

## 撤销操作与区域的关系

理解区域之后，撤销操作就很好记了：

| 命令 | 作用 | 影响的区域 |
|------|------|-----------|
| `git restore <file>` | 丢弃工作区的修改 | 工作区 |
| `git restore --staged <file>` | 取消暂存（但不丢工作区修改） | 暂存区 |
| `git reset --hard <版本>` | 回退到某个提交 | 工作区 + 暂存区 + 版本库 |

```
撤销工作区修改          git restore <file>
取消暂存（不丢修改）     git restore --staged <file>
回退版本（危险！）       git reset --hard <commit-hash>
```

## 常见问题

**Q：暂存区是不是远程服务器上的东西？**
A：不是。暂存区（Staging Area）是你本地机器上的一个区域。它在 `.git/index` 文件里记录了哪些文件要被提交。远程操作才涉及网络。

**Q：我不 add，直接 commit 可以吗？**
A：可以，但不好。`git commit -a -m "xxx"` 可以跳过 add 直接提交，但这意味着你没有机会选择性地暂存。不过在某些场景下（比如只改了一个文件）这是快捷的。

**Q：工作区干净的时候是什么意思？**
A：意味着工作区里没有"已修改但未暂存"的文件，所有修改都已经 commit 到本地版本库了。

**Q：暂存区的东西会不会丢？**
A：只要你不去动 `.git` 文件夹，Git 的数据不会丢。`.git` 就是你的本地版本库。

## 扩展：.git 目录是什么

当你在文件夹里执行 `git init` 后，会生成一个 `.git` 文件夹。这不是普通文件夹，而是 Git 的"数据库"。

```
.git/
├── objects/       # 所有 Git 对象（提交、文件快照）
├── refs/          # 分支和标签的指针
├── index          # 暂存区的索引文件
├── HEAD           # 当前指向的分支
└── config         # 当前仓库的配置
```

**⚠️ 不要手动修改 .git 文件夹里的内容**，除非你非常清楚自己在做什么。

## 本节总结

```
Git 三大区域：
  工作区  → 你正在编辑的地方（本地磁盘）
  暂存区  → 准备提交的文件清单（.git/index）
  版本库  → 提交后的历史记录（.git/objects）

文件状态：
  未跟踪 → 已修改 → 已暂存 → 已提交

核心命令：
  git add <file>              → 工作区 → 暂存区
  git commit -m "message"    → 暂存区 → 版本库
  git restore <file>          → 撤销工作区修改
  git restore --staged <file> → 取消暂存（不改工作区）
```

# Git 协作规范手册 — 设计方案

## 一、手册定位

- **使用者**：个人（我自己）
- **学习方式**：线性学习式，从头到尾有顺序阅读
- **阅读场景**：日常快速检索 + 系统性学习
- **最终形式**：项目文件夹中的一系列 .md 文档
- **操作系统**：Windows（Git Bash / WSL）/ macOS / Linux 通用
- **Git 平台**：GitHub

## 三、每个阶段的编写规范

### 页面结构

每个阶段一篇文章，结构如下：

```
### 本节目标
（3 句话说明学完这节能做什么）

### 核心概念
（用类比或生活化语言解释原理，不堆命令）

### 命令实战
（按步骤写的完整操作，带注释）

### 企业场景
（这个操作在真实工作中什么时候用）

### 常见问题
（FAQ / 踩坑点）

### 扩展命令
（可选，了解即可）
```

### 内容风格

- 用**我自己**的视角写，不是教程视角
- 每句话要有信息量，不废话
- 命令必须有注释，说明每一步在做什么
- 遇到可能出错的地方，用"⚠️ 注意"标注
- 跨平台差异用"🌍 跨平台"标注

## 五、设计原则

### 优先使用新版命令

| 旧命令（不推荐） | 新命令（优先使用） |
|----------------|------------------|
| `git checkout -- <file>` | `git restore <file>` |
| `git checkout -b <branch>` | `git switch -c <branch>` |
| `git checkout <branch>` | `git switch <branch>` |

### 命令注释规范

每个命令都要有注释，用 `#` 写在同一行或下一行：

```bash
git switch -c feature/user-auth    # 创建并切换到新分支（新命令）
git add -p                          # 交互式暂存，逐块选择
```

### 错误处理标注

```bash
⚠️ 如果出现冲突，Git 会提示你手动解决，不要慌——详见第 9 节
```

### 跨平台标注

```bash
🌍 Windows：~/.ssh（用户目录）
🌍 macOS/Linux：~/.ssh（同上）
```

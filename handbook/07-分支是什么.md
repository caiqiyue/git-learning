# 7. 分支是什么？为什么要分支？

## 本节目标

学完这节后，你会：
- 理解 Git 分支的本质（不是复制文件，是创建指针）
- 理解分支在团队协作中的价值
- 理解常见的分支策略

---

## 分支的本质

很多人以为分支是"复制一份代码"，这是一个误解。**分支的本质是：创建一个指向某个 commit 的指针**。

```
commit A (abc123)  ←─ HEAD ←─ main (指针)
```

当你创建一个新分支 `feature/login`，Git 做的只是：
1. 创建一个新的指针叫 `feature/login`
2. 指向当前 commit

```
commit A (abc123)
      ↑
    HEAD, main, feature/login  ← 三个指针指向同一个 commit
```

当你提交新代码时，只有当前分支指针移动：

```
commit A (abc123) ── commit B (def456)  ← HEAD 和 feature/login
      ↑
    main  ← main 还停在 commit A
```

**这就是 Git 分支"便宜"的根本原因：创建分支只是创建一个指针，不复制任何文件。**

## 为什么要用分支

### 场景 1：多人并行开发

**没有分支：**
```
小王和小明都在改 user.js
小王改完了发给你，你说"我这边也改了"
冲突！谁的版本是对的？
```

**有分支：**
```
小王 → feature/user-auth 分支
小明 → feature/payment 分支
两个人各改各的，互不干扰
最后通过 Code Review 合并到 main
```

### 场景 2：功能开发和线上 bug 同时存在

```
你正在开发 v2.0 新功能（开发到一半）
突然线上出 bug 需要立即修
```

**有分支：**
- 从 main 创建 `hotfix/critical-bug` 分支
- 在 hotfix 分支修 bug，并合并到 main
- 继续在原来的分支开发 v2.0

**没有分支：**
- 你得先把新功能代码"藏起来"或者直接回退
- 修完 bug 后再重写新功能，效率极低

### 场景 3：Code Review 机制

企业开发要求所有代码必须经过 Code Review 才能进入主分支。
- 你在 `feature/xxx` 分支开发
- 完成后创建 Pull Request（PR）
- 同事 Review 通过后，合并到 main
- main 分支始终保持稳定可发布状态

## 理解 HEAD 指针

**HEAD 是指向"当前分支最新提交"的指针。**

```
commit A ← HEAD ← main
```

当你执行 `git switch feature/login`，实际上是把 HEAD 指向了 `feature/login`：

```
commit A ← main
commit B ← HEAD ← feature/login
```

**⚠️ 理解这一点很重要：**
- `git checkout <commit-hash>` 会让 HEAD 处于"分离状态"（detached HEAD）
- 在 detached HEAD 状态下提交是危险的，因为没有任何分支指向它
- 新版本建议用 `git switch`，它会阻止你进入危险的 detached HEAD 状态

## 常见的分支策略

### GitFlow（适合中大型团队）

```
                        ┌─ hotfix/xxx（从 main 切出，紧急修复）
                       ↗              ↘
main ───●────────────────────────●─────────────────
              ↑                   ↑ merge
           release/              develop
              ↑ merge            ↑
                   ↖───────────●──── feature/xxx（从 develop 切出）
```

| 分支 | 职责 | 生命周期 |
|------|------|---------|
| main | 生产环境代码 | 永久 |
| develop | 开发主干，集成分支 | 永久 |
| release/x.y.z | 发布准备 | 短期 |
| feature/xxx | 功能开发 | 中期 |
| hotfix/xxx | 紧急修复 | 极短期 |

### Trunk-Based（适合小团队）

所有开发都在 main 或极短期分支（≤ 2 天）内完成：
```
main ────────────────────────────────
       ↖ feature/xxx (≤2天) ↗
```

适合：团队小（≤ 10 人）、CI/CD 成熟、快速迭代的产品。

## 分支命名规范

| 类型 | 命名格式 | 示例 |
|------|---------|------|
| 功能分支 | `feature/描述` | `feature/user-auth` |
| 修复分支 | `fix/问题描述` | `fix/login-timeout` |
| 热修复分支 | `hotfix/问题描述` | `hotfix/payment-crash` |
| 发布分支 | `release/x.y.z` | `release/1.2.0` |
| 实验分支 | `experiment/名称` | `experiment/new-algorithm` |

**企业实践：** 分支名中包含 ticket ID 便于追溯，例如：
```
feature/JIRA-1234-user-avatar-upload
hotfix/JIRA-5678-payment-null-exception
```

---

## 为什么企业强制要求用分支

1. **隔离开发环境**：每个人在自己的分支，互不干扰
2. **保护主分支**：main 分支是生产环境代码，不能随意修改
3. **可追溯**：谁在哪个分支改了什么，都有记录
4. **Code Review**：所有合并都要经过审查

## 常见问题

**Q：分支多了会不会占很多空间？**
A：不会。分支只是指针，不复制文件。Git 存储的是文件快照，不是差异。

**Q：可以创建多少个分支？**
A：理论上无限，但建议保持分支整洁，用完即删。

**Q：可以在一个分支里同时开发多个功能吗？**
A：不建议。一个分支应该只开发一个功能或做一件事。保持分支的单一职责。

**Q：hotfix 和 fix 有什么区别？**
A：hotfix 是从 main 切出来的紧急修复，通常绕过 develop 直接合并回 main。fix 一般是从 develop 切出的常规 bug 修复。

## 本节总结

```
分支本质：
  - 分支是指向 commit 的指针，不是文件副本
  - 创建分支只是创建指针，非常便宜
  - HEAD 指向当前分支的最新提交

分支的价值：
  - 多人并行开发，互不干扰
  - 功能开发和线上修复同时进行
  - 通过 PR 合并实现 Code Review

分支策略：
  - GitFlow：适合中大型团队，5 种分支类型
  - Trunk-Based：适合小团队，所有开发在 main 极短期分支

命名规范：
  - feature/xxx / fix/xxx / hotfix/xxx / release/x.y.z
  - 建议包含 ticket ID
```

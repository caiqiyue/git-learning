## 什么是保护分支

**保护分支是指设置了特殊规则的主分支，用于防止误操作和强制代码质量控制。**

当你对 main、master、develop 这些核心分支设置了保护规则后：
- 不能直接在这些分支上提交代码（必须通过 PR 合并）
- 不能强制推送（force push）
- 不能删除分支
- 必须满足指定条件才能合并（如通过 CI、通过 Code Review）

```
未保护：任何人都能直接 push 到 main，一行命令就能毁掉整个项目
已保护：push 到 main 被拒绝，必须走 PR → Review → 合并流程
```

## GitHub 保护分支配置

### 通过 Web 界面配置

1. 进入仓库 → Settings → Branches
2. 点击 "Add branch protection rule"（添加分支保护规则）
3. 填写规则信息：

```
Branch name pattern（分支名模式）:
  main                    ← 支持通配符，如 release/*

保护选项：
  ☑ Require pull request reviews before merging  ← 强制 Review
  ☑ Require status checks to pass before merging ← 强制 CI 通过
  ☑ Require branches to be up to date before merging
  ☑ Do not allow bypassing the above settings    ← 禁止管理员绕过
  ☑ Restrict who can push to matching branches   ← 限制推送权限
  ☑ Do not allow force pushes                    ← 禁止 force push
  ☑ Do not allow branch deletion                ← 禁止删除分支
```

### 通过 GitHub CLI 配置

```bash
# 安装 GitHub CLI（跨平台）
# Windows: 使用 winget install GitHub CLI
# macOS: brew install gh
# Linux: sudo apt install gh 或 sudo yum install gh

# 登录 GitHub
gh auth login

# 查看现有保护规则
gh api repos/{owner}/{repo}/branches/main/protection

# 添加保护规则（强制 PR + 至少 1 个 Review + 强制 CI）
gh api repos/{owner}/{repo}/branches/main/protection \
  --method PUT \
  --header "Accept: application/vnd.github.v3+json" \
  --field required_status_checks='{"strict":true,"contexts":["ci/check"]}' \
  --field enforce_admins=true \
  --field required_pull_request_reviews='{"required_reviewers":1}'

# 允许管理员强制推送（谨慎使用）
gh api repos/{owner}/{repo}/branches/main/protection/required_pull_request_reviews \
  --method PATCH \
  --field dismissal_restrictions='{}' \
  --field bypass_pull_request_allowances='{"admins":["{owner}"]}'
```

**注意：** 强制推送对其他人来说是灾难性的——如果有人 force push 了 main，你的本地 main 和远程 origin/main 就会分叉，再 push 会报错。保护分支就是用来阻止这种操作的。

## GitLab 保护分支配置

### 通过 Web 界面配置

1. 进入仓库 → Settings → Repository
2. 展开 "Protected branches"（受保护的分支）

```
Protected branches:
  main            ← 允许 Maintainers 合并， Developers 可以推送
  releases/*      ← 只允许 Maintainers 操作

权限级别：
  Allowed to merge（允许合并）:
    Maintainers / Developers / Masters
  
  Allowed to push（允许推送）:
    Maintainers / Developers / Masters
```

### 通过 GitLab API 配置

```bash
# 保护 main 分支
curl --request POST \
  --header "PRIVATE-TOKEN: your-gitlab-token" \
  --data "name=main" \
  --data "merge_access_level=40" \
  --data "push_access_level=40" \
  "https://gitlab.com/api/v4/projects/{project_id}/protected_branches"

# 参数说明：
# access_level: 10 =  Guest, 20 =  Reporter, 30 =  Developer, 40 =  Maintainer, 50 =  Owner
# 设置 40 意味着只有 Maintainer 及以上角色可以合并/推送
```

## 强制 Code Review

### 为什么需要强制 Review

```
没有 Review：
  我写的代码 → 直接合并 → 线上出问题
  代码质量靠自觉，质量参差不齐
  
有强制 Review：
  我写的代码 → PR → 同事 Review → 讨论改进 → 合并
  至少两个人看过代码，质量有保障，知识也共享了
```

### GitHub 配置强制 Review

```bash
# 强制至少 1 个 Reviewer 批准才能合并
gh api repos/{owner}/{repo}/branches/main/protection/required_pull_request_reviews \
  --method PUT \
  --field required_reviewers='[{"type":"User","reviewer":{"id":12345}}]'

# 或者通过界面：
# Settings → Branches → Add rule → Require pull request reviews before merging
# Required number of approving reviewers: 1（设置为 1 或更高）
```

### 设置特定人必须 Review

```bash
# 某些模块要求特定人 Review（如支付模块必须安全工程师 Review）
gh api repos/{owner}/{repo}/branches/main/protection/required_pull_request_reviews \
  --method PATCH \
  --field required_reviewers='[
    {"type":"User","reviewer":{"login":"security-expert"}},
    {"type":"Team","reviewer":{"slug":"security-team"}}
  ]'
```

**注意：** 如果设置了强制 Review 但 CI 失败，代码仍然无法合并。这是正确的行为——Review 通过了但代码有 bug，CI 会拦住。

## 禁止 Force Push

### Force Push 的危害

```bash
# 场景：如果有人在 main 上 force push 了
# 你的本地仓库状态
origin/main: A ← B ← C ← D  ←─ 你上次 fetch 的位置

# 有人 force push 了
origin/main: A ← X ← Y  ←─ main 被重写了！

# 你执行 git push
To github.com:owner/repo.git
 ! [rejected] main -> main (non-fast-forward)
 error: failed to push some refs
 hint: Updates were rejected because the tip of your current branch is behind
```

### 在保护分支中禁止 Force Push

```bash
# GitHub CLI
gh api repos/{owner}/{repo}/branches/main/protection \
  --method PUT \
  --field allow_force_push=false

# GitLab API
curl --request POST \
  --header "PRIVATE-TOKEN: your-token" \
  --data "name=main" \
  --data "force_push_access_level=0" \
  "https://gitlab.com/api/v4/projects/{project_id}/protected_branches"
```

**跨平台注意：** 在 Windows 上执行 curl 命令时，如果使用 Git Bash 或 WSL，语法一致。如果使用 CMD 或 PowerShell，需要对引号进行转义。

## 状态检查要求（CI/CD）

### 配置必须通过的 CI 检查

```bash
# 设置必须通过的 CI 检查（GitHub）
gh api repos/{owner}/{repo}/branches/main/protection/required_status_checks \
  --method PUT \
  --field strict=true \
  --field contexts='["ci/build", "ci/test", "ci/lint"]'
```

配置后，只有当这三个 CI 任务都通过时，才能合并 PR。

### 工作原理

```
PR 创建 → CI 运行 → 状态上报到 GitHub/GitLab
                              ↓
                    所有 checks passed?
                    ↓ yes         ↓ no
                 可以合并      合并按钮灰显
```

### 常见 CI 检查项

| 检查项 | 作用 | 失败后果 |
|--------|------|---------|
| `ci/test` | 运行单元测试 | 测试失败不允许合并 |
| `ci/lint` | 代码风格检查 | 不符合规范不允许合并 |
| `ci/build` | 构建产物 | 构建失败不允许部署 |
| `ci/e2e` | 端到端测试 | E2E 失败不允许合并 |
| `security/scan` | 安全扫描 | 有漏洞不允许合并 |

## 管理员如何绕过保护分支

### 正常情况管理员也不能绕过

```bash
# 管理员尝试 force push 到保护分支
$ git push origin main --force

! [remote rejected] main -> main (protected branch hook)
error: failed to push some refs
hint: Hookfsync prevented a forced push. Cannot force pushes to protected branches.
```

**注意：** 启用 "Do not allow bypassing the above settings" 后，即使你是仓库 Owner，也无法绕过保护规则。这是设计如此，目的是确保所有代码都必须经过审查流程。

### 需要绕过时的正确做法

1. **临时修改保护设置**（谨慎操作，需要记录）
   ```bash
   # 临时允许 force push（完成后记得改回来！）
   gh api repos/{owner}/{repo}/branches/main/protection \
     --method PUT \
     --field allow_force_push=true
   
   # 执行修复操作
   git push origin main --force
   
   # 立即恢复保护设置
   gh api repos/{owner}/{repo}/branches/main/protection \
     --method PUT \
     --field allow_force_push=false
   ```

2. **使用 git revert 而非 force push**
   ```bash
   # 错误提交后，正确做法是 revert 而不是 force push
   git revert abc1234    # 创建一个新提交，撤销 abc1234 的改动
   git push origin main  # 通过正常流程合并
   ```

3. **请求 Repository Admin 在界面临时关闭保护**
   - 进入 Settings → Branches
   - 临时关闭规则，执行操作，再重新开启

**跨平台注意：** 临时关闭保护是危险操作，建议在执行前通知团队，并在操作日志中记录原因。

## 常见保护分支策略

### 策略一：简单策略（小型团队）

```
核心分支：main（唯一受保护分支）
规则：
  ☑ 禁止 force push
  ☑ 禁止直接推送，必须通过 PR
  ☑ 至少 1 人 Review
```

适合：3-5 人小团队，信任基础好。

### 策略二：GitFlow 策略（中大型团队）

```
永久分支：
  main      → 始终受保护，禁止 force push，强制 CI + 2 人 Review
  develop   → 受保护，禁止 force push，至少 1 人 Review

短期分支（不需要保护）：
  feature/*、fix/*、hotfix/*、release/*
```

| 分支 | 保护级别 | 说明 |
|------|---------|------|
| main | 最高 | 生产环境代码，禁止任何 force push |
| develop | 高 | 开发主干，禁止 force push |
| release/* | 中 | 发布分支，发布完成后合并回 main |
| feature/* | 低 | 功能分支，开发完成后删除 |

### 策略三：多环境策略（微服务/多产品）

```
生产环境分支：
  production/main      → 最严格：强制 2 人 Review + 所有 CI 通过
  staging/main         → 严格：强制 1 人 Review + 所有 CI
  
开发分支：
  develop/main         → 中等：强制 1 人 Review + 基础 CI
  feature/*           → 不保护（短期分支）
```

### 策略选择建议

| 团队规模 | 产品类型 | 推荐策略 |
|---------|---------|---------|
| 1-5 人 | 内部工具/原型 | 简单策略 |
| 5-20 人 | 正常产品开发 | GitFlow 策略 |
| 20 人以上 | 微服务/多产品 | 多环境策略 |

## 本节总结

```
保护分支核心功能：
  1. 禁止直接推送 → 必须通过 PR
  2. 禁止 force push → 防止历史被覆盖
  3. 强制 Code Review → 至少一人看过
  4. 强制 CI 通过 → 代码质量门禁

管理员绕过：
  → 正常情况无法绕过
  → 临时修改保护设置（需谨慎）
  → 正确做法是用 revert 而非 force push

常见策略：
  → 简单：只保护 main
  → GitFlow：保护 main + develop
  → 多环境：按环境分级保护
```

# 5. 撤销修改：restore / reset / reflog

## 本节目标

学完这节后，你会：
- 知道工作区、暂存区、版本库各自的撤销方法
- 掌握 git restore（新版命令）替代旧版 checkout
- 理解 git reset 的三种模式（soft / mixed / hard）
- 学会用 reflog 在误操作后恢复

---

## 核心原则

撤销操作遵循一个简单原则：**在哪个区域出问题，就在哪个区域解决**

```
工作区出问题 → git restore <file>
暂存区出问题 → git restore --staged <file>
版本库出问题 → git reset [--soft|--mixed|--hard] <commit>
```

---

## 撤销工作区的修改（未 add，未 commit）

**场景：** 你改了一个文件，但还没 add，发现改错了，想全部撤销。

### 新版命令（Git 2.23+，优先使用）

```bash
git restore <file>      # 丢弃工作区的修改，恢复到暂存区/版本库的状态
```

### 撤销所有工作区修改

```bash
git restore .          # 当前目录所有文件
git restore --source=HEAD --staged .   # 或者：还原暂存区到 HEAD
```

**⚠️ 注意：这条命令会丢失你工作区的所有修改，谨慎使用！**

### 旧版命令（仍可用，但不推荐）

```bash
git checkout -- <file>    # 旧命令，不推荐
```

---

## 取消暂存（已 add，但未 commit）

**场景：** 你执行了 `git add .`，把一堆文件加入了暂存区，但现在发现有些不应该暂存，想取消暂存。

### 新版命令

```bash
git restore --staged <file>    # 取消暂存，但保留工作区修改
```

### 取消暂存所有文件

```bash
git restore --staged .        # 所有文件取消暂存
```

### 旧版命令（仍可用）

```bash
git reset HEAD <file>         # 旧命令
```

### 效果示意

```
操作前：
  工作区：已修改（红色）
  暂存区：有内容（绿色）

git restore --staged <file> 后：
  工作区：已修改（红色）← 保留
  暂存区：清空 ← 取消暂存了

git restore <file> 后：
  工作区：恢复成暂存区的样子 ← 丢弃修改
  暂存区：清空
```

---

## 回退版本（已 commit）

**场景：** 你已经 commit 了，但发现这次提交是错的，想回退。

### git reset 的三种模式

| 模式 | HEAD 位置 | 暂存区 | 工作区 | 危险程度 |
|------|----------|--------|--------|---------|
| `--soft` | 移到目标提交 | **保留** | **保留** | ⭐ 安全 |
| `--mixed`（默认） | 移到目标提交 | **清空** | **保留** | ⭐⭐ 安全 |
| `--hard` | 移到目标提交 | **清空** | **清空** | ⭐⭐⭐⭐ 危险 |

### 示例数据

```
假设当前提交历史：
  a1b2c3d (HEAD) feat: add feature C
  e4f5g6h         feat: add feature B
  i7j8k9l         feat: add feature A
```

#### --soft：温柔回退

```bash
git reset --soft HEAD~1    # 回退到上一个提交
```

效果：
- HEAD 指向上一个提交
- 暂存区**保留** feature C 的修改
- 工作区**保留**所有修改
- 工作区显示 feature C 的修改是"未暂存"状态

**使用场景：** 你想重新提交，但想修改 commit message 或者合并到下一次提交。

#### --mixed：保守回退（默认）

```bash
git reset HEAD~1           # 不加参数，默认是 --mixed
git reset --mixed HEAD~1
```

效果：
- HEAD 指向上一个提交
- 暂存区**清空**（feature C 的修改不在暂存区了）
- 工作区**保留** feature C 的修改
- 工作区显示 feature C 的修改是"未暂存"状态

**使用场景：** 你 commit 得太早，漏加了一个文件，想补上。

#### --hard：激进回退（危险！）

```bash
git reset --hard HEAD~1
```

效果：
- HEAD 指向上一个提交
- 暂存区**清空**
- 工作区**清空**（所有修改都没了！）
- 工作区干净

**⚠️ 这是最危险的命令，除非你确定不需要工作区的任何修改，否则不要用。**

---

## 精确回退到某个版本

```bash
# 回退到倒数第 N 个提交
git reset --hard HEAD~3

# 回退到指定 commit hash
git reset --hard a1b2c3d

# 查看某个提交的 hash
git log --oneline
# 输出：
# a1b2c3d feat: add feature C
# e4f5g6h feat: add feature B
git reset --hard e4f5g6h    # 回退到 feature B 的状态
```

## 撤销已 push 到远程的提交

⚠️ **这是团队协作中的危险操作，要非常谨慎。**

### 场景 A：不需要保留这个提交（直接覆盖）

```bash
git reset --hard <commit>
git push --force-with-lease origin <branch>
```

⚠️ 强制推送会覆盖远程历史，可能影响协作者。

### 场景 B：需要保留这个提交（安全的做法）

```bash
# 新建一个提交来"撤销"之前的改动
git revert <commit-hash>
git push origin <branch>
```

`git revert` 会创建一个新的提交，**内容正好是之前那个提交的相反操作**，但不改变历史。更安全，适合团队协作。

## reflog：救命机制（误操作恢复）

**这是 Git 最重要的"后悔药"。**

### 什么是 reflog

reflog（Reference Log）记录了 HEAD 的所有移动历史。即使你 `git reset --hard` 删除了 100 个提交，也能找回来。

### 查看操作历史

```bash
git reflog
# 输出：
# a1b2c3d HEAD@{0}: reset: moving to HEAD~3
# b2c3d4e HEAD@{1}: commit: feat: add feature C
# c4d5e6f HEAD@{2}: commit: feat: add feature B
# d5e6f7g HEAD@{3}: commit: feat: add feature A
# e6f7g8h HEAD@{4}: checkout: switching to main
```

### 误删分支后恢复

```bash
# 假设你不小心删除了 feature/payment 分支
git reflog
# 输出：
# a1b2c3d HEAD@{0}: checkout: deleting branch feature/payment (was b2c3d4e)

# 从 reflog 中找到分支最后一次 commit 的 hash
git switch -c feature/payment b2c3d4e    # 从该 hash 重建分支
```

### 误 reset --hard 后恢复

```bash
# 你执行了 git reset --hard HEAD~5，发现后悔了
git reflog
# 输出：
# f3a4b5c HEAD@{0}: reset: moving to HEAD~5
# a1b2c3d HEAD@{1}: commit: feat: add feature C  ← 想回到这里

git reset --hard a1b2c3d    # 恢复到 C 的状态
```

## 撤销命令速查表

| 场景 | 命令 |
|------|------|
| 丢弃工作区的修改 | `git restore <file>` |
| 取消暂存（不丢工作区修改） | `git restore --staged <file>` |
| 回退版本（保留工作区修改） | `git reset --soft HEAD~1` |
| 回退版本（保留工作区，取消暂存） | `git reset HEAD~1` 或 `git reset --mixed HEAD~1` |
| 回退版本（丢弃一切，危险！） | `git reset --hard HEAD~1` |
| 回退到指定 commit | `git reset --hard <commit-hash>` |
| 恢复误删的分支 | `git reflog` + `git switch -c <branch> <hash>` |
| 恢复误 reset 的提交 | `git reflog` + `git reset --hard <hash>` |
| 撤销已 push 的提交（安全方式） | `git revert <commit-hash>` |

## 常见问题

**Q：git restore 和 git checkout 有什么区别？**
A：`git checkout` 在旧版 Git 中既用于切换分支，又用于丢弃文件修改，语义不清晰。Git 2.23+ 引入了 `git restore`（专门恢复文件）和 `git switch`（专门切换分支），职责更清晰。**优先使用新命令。**

**Q：git reset 和 git revert 哪个更安全？**
A：`git revert` 更安全，因为它不改变历史，适合团队协作。`git reset` 会改写历史，只有在本地未 push 的情况下才用。

**Q：reflog 能恢复多久的历史？**
A：默认保留约 90 天。但如果你执行了 `git gc`（垃圾回收），可能会清理掉一些旧的 reflog 记录。

**Q：可以恢复已经删除但未 commit 的文件吗？**
A：如果你之前有 commit，可以从那个 commit 找回。如果从来没 commit 过（未跟踪文件），恢复不了。

## 本节总结

```
工作区撤销：git restore <file>
暂存区撤销：git restore --staged <file>
版本库回退：git reset [--soft|--mixed|--hard] <commit>

reset 三模式：
  --soft   HEAD移动，暂存区保留，工作区保留（最温柔）
  --mixed  HEAD移动，暂存区清空，工作区保留（默认）
  --hard   HEAD移动，暂存区清空，工作区清空（最危险！）

救命机制：
  git reflog   查看所有操作历史
  git reset --hard <hash>   从 reflog 恢复
```

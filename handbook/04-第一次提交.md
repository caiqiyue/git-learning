# 4. 第一次提交：init → add → commit → log

## 本节目标

学完这节后，你会：
- 完成一个完整的"修改 → 暂存 → 提交"流程
- 写出让别人看得懂的 commit message
- 查看提交历史，理解 git log 的输出

---

## 完整工作流程

这是 Git 最核心的操作循环，每个开发者每天都要做无数次：

```
工作区 → git add → 暂存区 → git commit → 版本库
```

---

## Step 1：创建或修改文件

```bash
# 进入项目目录
cd ~/my-project

# 创建或编辑一个文件
echo "# 我的项目" > README.md
echo "console.log('Hello Git')" > index.js
```

---

## Step 2：查看当前状态

```bash
git status
# 输出：
# Untracked files:
#    README.md
#    index.js
```

`git status` 是你每天都要用无数次的命令。**每次操作前先看一眼 status**，养成习惯。

---

## Step 3：暂存文件（git add）

### 暂存单个文件

```bash
git add README.md
```

### 暂存所有变更

```bash
git add .        # 暂存所有文件（包括新文件、修改、删除）
git add -A       # 同上，-A 是 --all 的缩写
```

### 交互式暂存（企业推荐！）

**⚠️ 这是企业中最常用的命令之一。**

```bash
git add -p
```

`-p` 是 `--patch` 的缩写，Git 会逐块（hunk）询问你要不要暂存：

```
Stage this hunk [y,n,q,a,d,s,e,?]?
```

| 按键 | 含义 |
|------|------|
| `y` | 暂存这一块 |
| `n` | 不暂存这一块 |
| `s` | 将当前块分割成更小的块 |
| `e` | 手动编辑这一块 |
| `q` | 退出 |

**为什么要用 add -p：**

你修改了一个文件，第 1-3 行是新功能（应该提交），第 4-5 行是调试代码（不应该提交）。

如果 `git add .`，两个改动会一起提交。用 `git add -p`，你可以只暂存第 1-3 行。

---

## Step 4：提交（git commit）

```bash
git commit -m "feat: add user login feature"
```

**⚠️ commit message 的重要性：**
- 好的 commit message 让你在 3 个月后仍能看懂当时的改动
- 团队协作中，commit message 是 code review 的重要参考
- 规范的 commit message 可以自动生成 CHANGELOG

### commit message 规范

详细内容见第 14 节，这里先记住基本格式：

```
<type>(<scope>): <subject>

type: feat / fix / refactor / test / docs / ...
scope: 受影响的模块（可选）
subject: 简短描述，用动词开头，不加句号
```

**例子：**

```bash
# ✅ 好的 commit message
git commit -m "feat(auth): add JWT token authentication"
git commit -m "fix(payment): resolve null pointer when amount is null"
git commit -m "docs(readme): update installation steps"

# ❌ 不好的 commit message
git commit -m "update"           # 太模糊
git commit -m "fix bug"          # 不知道修了什么
git commit -m "asdfasdf"         # ...
```

## Step 5：查看提交历史（git log）

### 基本用法

```bash
git log
# 输出：
# commit a1b2c3d4e5f
# Author: 张三 <zhangsan@example.com>
# Date:   Mon May 26 10:30:00 2026
#
#     feat(auth): add JWT token authentication
```

### 简洁格式（日常最常用）

```bash
git log --oneline
# 输出：
# a1b2c3d feat(auth): add JWT token authentication
# e4f5g6h fix(payment): resolve null pointer bug
# i7j8k9l docs(readme): update installation steps
```

### 图形化分支历史

```bash
git log --oneline --graph
# 输出更直观的分支图：
# * a1b2c3d feat(auth): add JWT token authentication
# * e4f5g6h fix(payment): resolve null pointer bug
# * i7j8k9l docs(readme): update installation steps
```

### 查看最近 N 条

```bash
git log --oneline -5    # 最近 5 条
```

## 完整演示

```bash
# 1. 创建项目并初始化
mkdir my-project && cd my-project
git init

# 2. 创建文件
echo "# My Project" > README.md
echo "console.log('Hello')" > index.js

# 3. 查看状态
git status

# 4. 交互式暂存（推荐）
git add -p    # 逐块选择暂存

# 或者简单暂存
git add .

# 5. 提交
git commit -m "feat: initial project setup"

# 6. 查看历史
git log --oneline

# 7. 再次修改后提交
echo "console.log('new feature')" >> index.js
git add .
git commit -m "feat: add new feature output"

# 8. 查看最新历史
git log --oneline -5
```

## 常见问题

**Q：提交信息写错了，能改吗？**
A：可以，但只限于本地未推送的提交：

```bash
# 修改最后一次提交信息
git commit --amend -m "新的提交信息"
```

⚠️ 如果已经 push 到远程，不要 amend（会改变 commit hash）。

**Q：想看某个特定文件的历史？**

```bash
git log --oneline -- index.js
# 或者
git log -p -- index.js    # 加上 diff
```

**Q：不小心 commit 了不该提交的内容怎么办？**

```bash
# 回退到上一次提交（工作区会保留修改）
git reset --soft HEAD~1
# 然后可以重新 add 和 commit
```

**Q：git add . 和 git add -A 有什么区别？**
A：
- `git add .` 在某些旧版 Git 中不包括删除操作
- `git add -A`（或 `--all`）包括所有变更（新增、修改、删除）

## 本节总结

```
工作流程：
  git status              # 先看状态
  git add <file>         # 暂存文件（或 git add . 暂存所有）
  git add -p             # 交互式暂存（推荐！逐块选择）
  git commit -m "message"  # 提交
  git log --oneline      # 查看历史
  git status             # 验证提交成功（工作区应该干净）

⚠️ 好习惯：
  - 每次操作前先 git status
  - 优先用 git add -p 而不是 git add .
  - commit message 要有意义，不是随意写的
```

# 6. 暂存工作进度：git stash

## 本节目标

学完这节后，你会：
- 理解为什么需要 stash（临时中断工作）
- 掌握 stash 的所有常用命令
- 在真实工作场景中使用 stash

---

## 什么是 stash

**stash = 临时存放工作进度**

当你正在写代码，改了一半，突然要去做别的事情（比如修 bug、切换分支），但又不想 commit 半成品代码时，可以用 `git stash` 把当前工作进度"藏起来"，等回来再恢复。

---

## 场景导入

**典型场景：**

你正在 `feature/payment` 分支上开发支付功能，代码改了 60%，突然产品经理说：

> "线上出了一个严重 bug，必须今天修完，你先停下手头的功能。"

**问题来了：**
- 直接切换分支？不行，未提交的修改会带过去
- commit？不行，代码只完成 60%，commit 信息没法写
- 放弃修改？不行，半天的活白干了

**解决方案：git stash**

---

## 基本命令

### 暂存当前工作进度

```bash
# 暂存当前所有修改（带描述）
git stash push -m "wip: feature/payment - 60% done, payment controller"

# 或者简单写法
git stash -m "wip: feature/payment"
```

### 查看暂存列表

```bash
git stash list
# 输出：
# stash@{0}: On feature/payment: wip: feature/payment
# stash@{1}: On develop: wip: fix memory leak
```

### 恢复暂存并删除记录

```bash
git stash pop    # 恢复最近一次 stash，并删除记录
```

### 恢复暂存但保留记录

```bash
git stash apply    # 恢复最近一次 stash
git stash apply stash@{0}    # 恢复指定的 stash（stash@{1} 等）
```

### 删除 stash 记录

```bash
git stash drop stash@{0}    # 删除指定的 stash
git stash clear              # 清空所有 stash
```

## 完整使用流程

```bash
# 1. 你正在开发新功能
echo "payment logic" >> payment.js
git add .

# 2. 突然要修 bug
git stash -m "wip: payment feature in progress"

# 3. 切换到 bugfix 分支修 bug
git switch main
git switch -c hotfix/urgent-bug
# ... 修 bug，commit ...
git push -u origin hotfix/urgent-bug

# 4. bug 修完，PR 合并后，回到原分支继续开发
git switch feature/payment
git stash pop    # 恢复之前的工作进度

# 5. 继续开发
echo "complete payment" >> payment.js
git add .
git commit -m "feat(payment): complete payment integration"
```

## 常见场景

### 场景 1：需要拉取最新代码，但工作区有修改

```bash
# 你改了一些东西，但还没 commit
git stash
git pull --rebase origin main    # 拉取最新代码
git stash pop                     # 恢复工作进度
```

### 场景 2：切换分支前暂存

```bash
git stash
git switch other-branch
# ... 做其他事情 ...
git switch feature/payment
git stash pop
```

### 场景 3：多人同时开发，rebase 后恢复工作

```bash
# 你的分支和远程分支"分叉"了
git fetch origin
git reset --hard origin/feature/payment

# 恢复之前 stash 的工作
git stash pop
```

## stash 的高级用法

### 暂存未跟踪的文件（新增的文件）

```bash
git stash -u    # -u = --include-untracked
```

### 查看 stash 的具体内容

```bash
git stash show -p    # 查看最近一次 stash 的详细 diff
git stash show -p stash@{0}    # 查看指定 stash
```

### 暂存已暂存的文件

默认 `git stash` 只会暂存已跟踪文件的修改，不会暂存已暂存（staged）文件。如果需要：

```bash
git stash -S    # -S = --staged
```

### 暂存特定的某个文件

```bash
git stash push -m "wip: payment file" payment.js
# 只会暂存 payment.js 的修改
```

## 常见问题

**Q：stash 和 commit 有什么区别？**
A：commit 是永久记录，会进入版本历史。stash 是临时的"剪贴板"，帮你暂时搁置工作，不污染提交历史。

**Q：stash 的内容存在哪里？**
A：`.git/refs/stash` 和 `.git/objects/` 中，是本地数据，不会 push 到远程。

**Q：stash 可以 push 到远程吗？**
A：不行。stash 是本地临时存储，不会同步到远程服务器。

**Q：rebase 过程中能用 stash 吗？**
A：可以，但建议先完成 rebase，或者 stash 后再 rebase。

**Q：如果 stash pop 的时候有冲突怎么办？**
A：Git 会报冲突，你手动解决冲突后，用 `git add .` 标记解决，然后 Git 会帮你完成 stash pop（这个 stash 记录会被删除）。

**Q：可以暂存多个工作进度吗？**
A：可以。每个 `git stash` 会创建一个新的 stash，按时间顺序排列：`stash@{0}` 是最近一次，`stash@{1}` 是倒数第二次。

---

## 警告

⚠️ **stash 不等于备份。** 它只是临时存放工作进度。如果你在 stash 后删除了 `.git` 文件夹，stash 也会丢失。

⚠️ **长时间不清理的 stash** 可能导致冲突越来越多。所以用完 stash 后，记得 `git stash pop` 或 `git stash drop` 清理掉。

```
暂存工作：

恢复工作：
  git stash pop                    # 恢复并删除记录（常用）
  git stash apply                  # 恢复但保留记录
  git stash apply stash@{0}        # 恢复指定 stash

查看和删除：
  git stash list                   # 查看所有 stash
  git stash show -p                # 查看 stash 内容
  git stash drop stash@{0}        # 删除指定 stash
  git stash clear                  # 清空所有 stash

使用场景：
  - 开发到一半，需要切换分支修 bug
  - 需要拉取最新代码，但工作区有未提交的修改
  - rebase 前暂存工作进度
```
